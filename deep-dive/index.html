A personal work OS · 0.2

An autonomous engineer, built on a reducer, a queue, and a vector index.

TARS is a Personal AI Work OS for a senior IC at Microsoft. It watches your Azure DevOps queue, plans the work, writes the code in an isolated git worktree, reviews its own diff, waits for CI, and opens a draft PR — while keeping humans in the loop at every gate. This page is the guided tour — product, then architecture, then the non-obvious parts.

01what it is, concretely

Four ideas that make TARS different.

Most "AI assistants" are a chat box with tools. TARS is closer to a build system with opinions — a state machine for your actual work, persistent memory across sessions, and a clear separation between the things an agent does and the things TARS tracks.

01 · Autonomous, not chatty Work items move without you. A work-item scanner polls ADO every five minutes. Classified "code-change" tasks go to a planner, then a coding agent, a review agent, a verify agent, and a PR submitter — each an event in an append-only log. You approve gates, not each step.
02 · Pure-reducer FSM The state machine is a function. reducer(state, event) → {nextState, effects[]}. No side-effects in the transition — effects are data, enqueued separately. Worker crashes don't corrupt state; replay the event log and you're back.
03 · Memory, not cache Every agent output is a record. Briefings, standups, notes, plans — all land in one records table. Chunked, embedded with text-embedding-3-small, retrieved with cosine similarity plus a 1.5× recency boost for items under 14 days old.
04 · MCP as tracking bus Agents own their tools. TARS owns the ledger. The coding agent uses the repo's own git/npm/az CLIs via Claude Code's native tools. TARS's MCP server exposes only tracking verbs — record_branch, record_pr, update_progress — so the dashboard reflects reality without couplings leaking.

03the agent contract

Every agent is the same eleven lines.

A daily briefing, a monthly retro, a background ADO sync, a one-on-one prep — all implement the same abstract class. Context is built once by the orchestrator; the agent just gets a string and returns a string. validate() is the only contract they can fail.

packages/agents/src/shared/base.ts L1–L11 
export abstract class AgentDefinition {
  abstract readonly agentName: string;
  abstract readonly recordType: string;
  abstract readonly epoch: string;

  /** Skip full context (ADO, M365, embeddings) for lightweight agents */
  readonly skipContext?: boolean;

  abstract execute(context: string, data: Record<string, unknown>): Promise<string>;
  abstract validate(output: string): string;
}

What each field earns.

agentName — the key in the registry (packages/agents/src/shared/registry.ts:18–34). 15 instances, looked up by string. It's how schedulers enqueue without holding a reference.

recordType — the slot in Postgres where output lands. Unique index on (type, sourceAgent, dateKey) enforces one canonical output per agent per day.

epoch — "daily", "weekly", "monthly", "adhoc", "system". Drives retention, rollup, and which long-term context gets folded into the next call.

skipContext — the escape hatch. The knowledge indexer doesn't need a briefing of the world; it needs raw rows. Skipping saves the embedding round-trip.

validate() — executed before anything hits the DB. A daily briefing under 100 characters is rejected; a daily note under 200 is rejected. The one guard-rail an agent must pass.

05unified memory

One table. Vectors on the side.

TARS doesn't have "memory" as a separate subsystem. It has records — one table where every agent writes its output — and embeddings, a per-chunk RAG index. A recency boost nudges younger memories up the ranking.

SELECT
  e.chunk_text,
  r.type,
  r.date_key,
  r.id as record_id,
  (1 - (e.embedding <=> $1::vector)) *
    CASE WHEN r.created_at > NOW() - INTERVAL '14 days'
         THEN 1.5 ELSE 1.0 END
    AS score
FROM embeddings e
JOIN records r ON e.record_id = r.id
WHERE r.status IN ('approved', 'published')
ORDER BY score DESC
LIMIT $2

06orchestration

Redis is the spine.

Everything that runs goes through a BullMQ queue. The scheduler enqueues, workers consume, effects enqueue again. No agent is executed directly from an API route or a cron tick.

09insights from the codebase

The non-obvious load-bearing parts.

Things you'd only notice by reading the code or chasing a production bug. Each one earns its keep.

I · 01

Idempotency keys are run-scoped, never task-scoped.

Format: <kind>:<runId>, e.g. discover:46d6d485. A conditional unique index — UNIQUE (idempotency_key) WHERE idempotency_key IS NOT NULL — silently drops replays. Writing task-scoped keys collides across runs of the same ADO item, which happened once and never again.

schema/executor-events.ts:31–33 force-enter-work-item/route.ts:94
I · 02

The FSM is a function, not a library.

reducer(state, event) → {nextState, effects[]}. No Redis calls, no DB calls, no awaits. Effects are pure data — {kind:"dispatch_coding", runId, …}. The worker applies them after state is computed, so a crash mid-effect leaves a consistent event log to replay.

executor/fsm.ts:9–46 executor/effects.ts:9–70
I · 03

MCP exposes verbs, not capabilities.

The coding agent's MCP server offers update_progress, record_branch, record_pr, update_ado_status. Nothing that does git. The agent uses the repo's own CLAUDE.md skills for branching, committing, pushing. TARS tracks what happened; it doesn't impose how.

interactive/coding/mcp.ts:44–199
I · 04

update_ado_status refuses to close.

The tool enumerates forbidden states — Closed, Done, Resolved, Completed, Removed — and rejects them at the MCP boundary. A hallucinating agent can't mark a work item done; that's a human's job via the dashboard.

interactive/coding/mcp.ts:174–199
I · 05

Recency as a multiplier, not a filter.

Semantic search doesn't cap context to recent docs — it multiplies cosine similarity by 1.5× when the record is under 14 days old. A stale-but-relevant monthly note still surfaces; a current-and-relevant one surfaces first.

retrieval/semantic-search.ts:33–34
I · 06

Cost and iteration caps are gates, not kills.

If review loops exceed 3 iterations or spend > $20, the state machine doesn't fail — it transitions to awaiting-review-caps-gate, opens a human signal, and waits. You can approve a budget top-up and continue, or abandon. Defaults live in one place.

constants.ts:187–196 executor/fsm.ts:365–389
I · 07

Terminal statuses live in two places, on purpose.

CODING_TASK_TERMINAL_STATUSES = ["done","failed","abandoned","blocked"] appears in the constants file and in the partial-index WHERE clause of the worktree-pool slot index. A comment warns: if you drift them, pool queries silently seq-scan. The coupling is known, named, and load-bearing.

constants.ts:54–64 schema/coding-agent-tasks.ts
I · 08

The voice service serializes ONNX inference on purpose.

Concurrent Kokoro TTS inferences thrash CPU, making each ~3× slower. A single-request semaphore gates real-time inference; pre-cached openers ("Humor setting at seventy five percent. Also, working on it.") bypass it entirely. Observed latency wins more than parallelism would.

voice-service/server.ts:337–365
I · 09

Windows az.cmd quoting is a module of its own.

Node's execFile with shell:true on Windows re-splits arguments through cmd.exe. Values with spaces, quotes, or JSON vanish. exec-cli.ts pre-quotes every argument; Unix gets a thin pass-through. Ten az call sites fixed in one sweep.

packages/shared/src/exec-cli.ts:32–80
I · 10

Workers pause on flags, not kills.

Six Redis keys — tars:flags:review-paused, coding-paused, pr-handler-paused, etc. — let you freeze a single phase mid-pipeline without touching BullMQ. Paused jobs throw "phase paused — retry later"; BullMQ's exponential backoff handles the rest.

constants.ts:175–185
