What is TARS

A senior IC's work operating system, assembled out of agents.

TARS is an autonomous, queue-driven platform that watches your ADO board, drafts your standups, plans your features, and writes the first PR — all under human review, none of it auto-published.

Built for one user — a Microsoft IC. The whole thing runs on a laptop or DevBox, talks to ADO, M365, and Azure, and calls Claude through the local claude login session — no metered API key.

004 · Rules of the Road
005 / 015

Three Architectural Rules

The invariants that keep TARS legible.

i.
No layer
skips a layer.

UI never reaches into Azure or M365. Every cross-boundary call goes through a queue or an L3 query helper — observable, retryable, mockable.

ii.
Every agent
through BullMQ.

No direct execution from cron, API routes, or chat sessions. Schedulers enqueue; workers execute. Backpressure, retries, and pause flags come for free.

iii.
Human-in-the-loop
always.

Agent outputs land as draft. You approve. Coding agents always create draft PRs; the executor opens approval gates at every cap. No auto-publish, no auto-merge.

The Idea That Holds Everything Together

Plan · Code · Review · Verify · loop.

The whole executor is one diagram: a plan, a coder, two auditors, and a feedback arrow that costs money.

  • a.
    Forward path — solid amber arrows. Plan → code → review → verify → submit.
  • b.
    Feedback path — dashed rose arrows. Blockers re-enter the coder with a directive.
  • c.
    Caps — review and verify each cap at 3 iterations or $20; over caps, a gate opens.
  • d.
    After submit — control passes to the PR Handler FSM, which keeps watching the PR until merge.

009 · Feature · Executor v0.2
010 / 015

Feature 4 of 5

Four queues, one event log,
one terminal state.

The executor isn't a script — it's an event log replayed through a pure reducer. State is derivable from the log.

  • →
    Phase split for backpressure — review and verify get concurrency 2 so they don't stall the FSM.
  • →
    Idempotency key per event (<kind>:<runId>) — re-running an effect is a no-op.
  • →
    FSM single-threaded — concurrency 1 on the executor queue means events apply in order.
  • →
    30 min timeout per iteration · $20 / 3 iters caps per phase.

010 · Executor · Feedback Mechanics
011 / 015

How The Loop Closes — Or Doesn't

Every iteration must justify its cost.

Failure has two shapes: try again (under cap, with directive) and ask the human (cap exceeded, gate opens).

  • →
    Loopback directive: blockers from review or verify are formatted into a follow-up prompt for the coder. The next iteration sees what the auditor said.
  • →
    Cost budget: totalCostUsd sums coder + reviewer + verifier across all iterations. Cap defaults are conservative — easy to lift via gate top-up.
  • →
    Gate decay: an unanswered approval gate decays after 24h via the tars:gate-decay background queue.
  • →
    Reverify: the PR Handler may re-enter the coder post-merge if a reviewer pushes back. Same idempotent FSM, separate run.

011 · Feature · Custom Agents
012 / 015

Feature 5 of 5

Add a new agent without touching code.

The same orchestrator runs both built-in and user-defined agents. The only difference is whether the prompt lives in .ts or in a Postgres row.

  • →
    Two output modes. Single record (one markdown summary) or list of items (each with an optional action proposal).
  • →
    Dynamic schedule. The cron registrar polls the table every 5 minutes — your edits propagate without restart.
  • →
    Action proposals for list-mode items connect into the L4 integrations layer (e.g. queue a reply, post to Engage).
  • →
    Validation still applies — Zod schemas guard the boundary into the database. Bad output, no record.

Three Things to Take Home

If you remember nothing else.

01.
Queues are
the boundary.

Every cross-boundary call goes through BullMQ. That's where retries, pauses, observability, and idempotency live. Skip the queue and you skip all of them.

02.
The executor is
an event log.

Pure reducer over an immutable event stream. State derives from the log. Crash mid-iteration, restart, replay — same answer.

03.
Drafts only.
You're the gate.

Records draft. PRs draft. Caps trigger gates. The system never publishes without you — and the cost of being wrong is always one approval away.
